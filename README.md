# 7조 — B+트리 인덱스 기반 SQL 조회 성능 개선
 
## 프로젝트 개요
 
기존 SQL 처리기의 `SELECT ... WHERE ...` 조건 처리 방식을 **선형 탐색(O(n))** 에서 **B+트리 인덱스(O(log n))** 로 개선하여 대용량 데이터 조회 성능을 향상시킨 프로젝트입니다.
 
## 배경
 
이전 주차에 구현한 SQL 처리기는 `WHERE` 조건이 들어오면 테이블을 처음부터 끝까지 확인하는 **선형 탐색** 방식을 사용했습니다. 데이터가 적을 때는 문제가 없지만, **100만 건** 수준의 대용량 데이터에서는 조회 시간이 크게 증가하는 한계가 있었습니다.
 
## 핵심 구현 내용
 
### 메모리 기반 B+트리 인덱스
 
이번 구현은 디스크 기반 영구 인덱스가 아닌 **메모리 기반 인덱스**입니다.
 
1. `SELECT` 쿼리가 들어오면 `players.csv`를 메모리에 로드
2. 로드된 데이터를 기준으로 `id`, `game_win_count` 컬럼에 대한 B+트리 구성
3. B+트리의 value로 파일 offset이 아닌 **row_index**를 저장
4. key 검색 시 파일을 다시 읽지 않고, 메모리에 올라와 있는 row에 **즉시 접근**
### 인덱스 대상 컬럼
 
| 컬럼 | 특징 | 목적 |
|---|---|---|
| `id` | 고유값 (Unique Key) | 단건 조회 성능 최적화 |
| `game_win_count` | 중복값 (Duplicate Key) | 중복 key 처리 학습 및 범위 조회 |
 
`id` 외에 `game_win_count`에도 인덱스를 구성한 이유는, 고유값 조회뿐 아니라 **중복 key 처리**를 직접 다뤄보기 위함입니다.
 
## 성능 비교
 
### 단건 조회 — `WHERE id = 500000`
 
| 방식 | 시간 복잡도 | 결과 |
|---|---|---|
| 선형 탐색 | O(n) | 느림 |
| B+트리 | O(log n) | **매우 빠름** |
 
- `id`는 고유값이므로 B+트리 인덱스의 효과가 가장 크게 나타남
- 최초 조회 시에는 CSV 파일을 메모리에 올리는 과정이 포함되어 양쪽 모두 시간이 소요됨
- 두 번째 조회부터 B+트리 쪽의 성능 우위가 명확하게 확인됨
### 범위 조회 — `WHERE game_win_count > 50`
 
| 방식 | 시간 복잡도 | 결과 |
|---|---|---|
| 선형 탐색 | O(n) | 느림 |
| B+트리 | O(log n + k) | 성능 차이 감소 |
 
- 하나의 key에 여러 row가 연결되는 중복 key 상황
- 조건에 맞는 row가 많아질수록 결국 읽어야 하는 데이터가 증가
- B+트리의 성능 이점이 단건 조회에 비해 줄어듦
## 결론
 
> **B+트리는 조회 조건에 따라 큰 성능 이점을 줄 수 있지만, 모든 상황에서 유리한 것은 아니다.**
 
- **단건 조회** (`WHERE id = ?`) → B+트리가 매우 유리
- **다건 범위 조회** (`WHERE col > ?`) → 결과 row 수가 많아지면 성능 차이가 줄어듦
## 데모 참고사항
 
- 1,000,000건의 데이터를 SQL INSERT로 하나씩 넣는 과정은 시간상 생략
- 미리 생성해둔 `players.csv` 파일을 사용하여 데모 진행
